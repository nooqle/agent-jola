# Agent Jola 格子爆破对战方案第三方评审

评审对象：[agent-jola-game-design.md](./agent-jola-game-design.md)

评审日期：2026-05-15

## 1. 总体结论

方案方向可行，但当前文档仍是“概念设计 + 技术草案”，还不是可以直接拆 Sprint 的工程规格。

核心架构判断是正确的：实时游戏主循环由本地确定性引擎负责，LLM/OpenClaw 只处理高层策略解释，不进入逐帧动作链路。这一点能避开延迟、稳定性和成本问题。

主要风险不在“能不能做”，而在以下四点：

- MVP 范围偏大：OpenClaw、Web Chat、多人、Agent、复盘、前端展示都被放进第一阶段。
- 规则细节不足：移动、速度、碰撞、爆炸结算顺序、同格冲突、随机种子等还没有精确定义。
- Agent 规格还偏抽象：危险图、路径规划和目标评分已有方向，但缺少时序模型、输入输出协议和失败处理。
- 对话控制边界需要收紧：自然语言可以改策略，但哪些指令允许、何时生效、谁有权限控制哪个 Agent 尚未定义。

建议下一步不是直接写前端，而是先把 `packages/core` 的纯规则引擎和无 UI 模拟跑通。只要核心引擎和 Agent simulator 稳了，前端和 OpenClaw 接入都可以较低风险地往上加。

## 2. 关键依赖清单

### 2.1 产品决策依赖

这些不是技术包，而是会阻塞架构和工期的产品决定：

1. 第一版是纯 Agent 对战，还是允许玩家手动控制一个角色。
2. 被爆炸命中是直接出局，还是进入“困住/可救援”状态。
3. 第一版是否必须接入 OpenClaw，还是先用本地 Web Chat 或调试面板模拟。
4. 对战是本地单房间演示，还是支持多人在线房间。
5. 一局游戏的主要价值是娱乐观战，还是训练/评估 Agent。
6. 美术资产使用占位符、像素风，还是定制完整视觉。
7. 是否需要长期记忆，例如记住用户偏好的打法。

其中 1、2、3、4 是开工前必须定下来的阻塞项。

### 2.2 工程依赖

建议依赖如下：

- Node.js：服务端、工具链和前端构建运行时。
- TypeScript：共享规则、协议和 Agent 逻辑。
- 包管理器：建议 `pnpm`，便于 monorepo 管理。
- Vite：前端开发与构建。
- React：房间 UI、策略面板、日志面板。
- Phaser 3 或 PixiJS：游戏地图和角色渲染，二选一即可。
- WebSocket 库：`ws` 或 Socket.IO，二选一即可。
- Vitest：核心规则、Agent planner 和协议测试。
- Schema 校验：建议引入 `zod` 或同类库，用于策略指令、WebSocket 消息和 API 输入校验。
- ESLint/Prettier：代码一致性。

MVP 不建议一开始引入数据库、Redis、队列、ORM、账号系统或复杂状态同步框架。

### 2.3 AI 与对话依赖

- OpenClaw tool 或 skill：用于把用户自然语言送入策略解释层。
- LLM provider：需要明确用哪个模型、由谁提供 key、调用预算和失败策略。
- 策略 schema：Agent 可接受的结构化策略字段。
- 指令解析器：可以先用规则/关键词 + LLM fallback，不必全量依赖 LLM。
- 权限模型：必须知道某个用户能控制哪个 match 和哪个 agent。

### 2.4 玩法内容依赖

- 地图模板或地图生成器。
- 初始出生点规则。
- 软墙分布规则。
- 道具掉落概率。
- Agent 初始属性和平衡参数。
- 占位角色、障碍、水泡、爆炸、道具素材。
- 可选音效。

### 2.5 运行与验证依赖

- 无 UI simulator，用于快速跑 1000 局以上的 Agent 对战。
- 决策日志格式。
- 固定随机种子机制。
- 性能指标：tick 耗时、planner 耗时、消息延迟。
- 回归测试地图集合。

## 3. 依赖关系与推荐实现顺序

推荐依赖链：

```text
规则模型与 tick 结算
        ↓
无 UI match simulator
        ↓
危险图与安全路径
        ↓
基础 Agent planner
        ↓
WebSocket 协议
        ↓
前端渲染
        ↓
策略解释层
        ↓
OpenClaw 接入
```

这条链的原因是：Agent、前端和 OpenClaw 都依赖稳定的核心规则。如果先做 OpenClaw 或 UI，会很快遇到“游戏语义还没定”的返工。

## 4. 完整性评审

### 4.1 已经覆盖得比较完整的部分

- 大模型不逐帧控制，职责边界清楚。
- 服务端权威的方向正确。
- Agent 需要危险图、安全路径、放泡前逃生检查，这几个关键点正确。
- OpenClaw 接口只更新策略，不直接发动作，这个边界正确。
- 测试方向覆盖了单元测试、模拟测试、集成测试和可解释性测试。
- IP 风险已经明确提出。

### 4.2 仍缺失的关键规格

#### Tick 与移动模型

文档提到固定 tick 和 `speed`，但没有定义速度如何影响格子移动。

必须补充：

- 玩家是否永远按格移动，还是有像素级过渡。
- `speed` 是缩短移动 tick，还是增加每 tick 位移。
- 移动中途是否可以转向。
- 两个玩家同时进入同一格如何处理。
- 玩家和水泡同格的阻挡规则，例如刚放泡后是否允许走出。

#### 爆炸结算顺序

文档有水泡和爆炸规则，但结算顺序还不够精确。

必须补充：

- 同一 tick 中，移动先结算还是爆炸先结算。
- 放泡和被炸如果发生在同一 tick，谁优先。
- 连锁爆炸是否立即在同一 tick 传播。
- 爆炸持续多少 tick。
- 道具是否会被爆炸摧毁。

#### 协议与状态同步

目前只写了 WebSocket 状态同步，没有定义消息协议。

必须补充：

- 客户端发什么消息。
- 服务端广播 full snapshot 还是 delta event。
- 断线重连如何恢复。
- 观战客户端是否能接收隐藏信息。
- 客户端是否做预测，还是只渲染服务端状态。

#### Agent 时序模型

文档写了 Agent 动作可提前一个 tick 计算，但没有定义 planner 的执行预算。

必须补充：

- Agent 每多少 tick 决策一次。
- 决策超时如何 fallback。
- planner 是否能读取完整未来信息。
- 对手未来动作如何估计。
- 上一次路径是否缓存，何时重算。

#### 策略解释层

自然语言到策略 patch 仍偏概念。

必须补充：

- 支持的指令类别白名单。
- 不支持指令如何回复。
- 指令置信度低时是否追问。
- 临时指令和全局风格冲突时谁优先。
- 策略 patch 是否有审计日志。

#### 安全与权限

文档没有涉及“谁能控制哪个 Agent”。

必须补充：

- match ownership。
- agent ownership。
- OpenClaw 来源签名或 token。
- 用户是否可以在对局中控制对手 Agent。
- API 限流，避免刷指令影响实时服务。

#### 评估指标

成功标准还偏主观。

建议量化：

- Agent 自杀率低于某阈值。
- 1000 局模拟无死循环或异常。
- 单 tick 结算 P95 小于指定毫秒数。
- planner P95 小于指定毫秒数。
- 策略变更后 N 秒内行为指标发生可观察变化。

## 5. 矛盾与不一致

### 5.1 “玩家不直接操控角色”与 `human` 类型冲突

原文开头写“玩家不直接操控角色”，但数据模型里有 `type: "human" | "agent"`，后续问题里又询问是否允许人类手动控制角色。

这不是致命矛盾，但说明 MVP 模式未定。建议明确：

- MVP-A：纯 Agent 对战，玩家只通过对话调策略。
- MVP-B：1 人手动 + N Agent。

推荐 MVP-A。

### 5.2 `kick` 同时出现在第一版道具和第一版可以没有

道具章节把 `kick` 列入第一版建议道具，但又说可放到第二阶段；MVP 非目标里也写“踢泡”可以没有。

建议从 MVP 删除 `kick`。踢泡会显著增加路径预测和危险图复杂度，不适合作为第一阶段。

### 5.3 “多人或多 Agent 对战”与“单房间 MVP”边界不清

产品目标说支持多人或多 Agent 对战，MVP 写单房间 2-4 个玩家或 Agent。这里容易被理解成要同时支持人类多人在线和 Agent 多人对战。

建议改成：

- MVP：单房间，本地或单服务实例，2-4 个 Agent，可观战。
- 第二阶段：人类玩家加入。
- 第三阶段：多人在线房间。

### 5.4 “OpenClaw 必须接入”与“Web Chat 模拟”优先级不清

文档目标中 OpenClaw 是核心入口，MVP 中也列为必须项，但后续问题又问是否首发接入。

建议明确为：

- 第一阶段先做 `/strategy` HTTP API + 本地调试面板。
- OpenClaw 作为同一 API 的外部调用方接入。

这样不会把游戏核心开发阻塞在 OpenClaw 集成细节上。

### 5.5 `riskTolerance` 与“永远避免必死动作”需要定义边界

文档说永远避免已知必死动作，除非策略显式允许高风险且收益极高。这会让 `riskTolerance` 的语义变得含糊。

建议分成两层：

- 硬约束：已知必死动作禁止。
- 软偏好：风险高但非必死的动作由 `riskTolerance` 调整。

MVP 不应允许 Agent 选择已知必死动作。

### 5.6 `trapped` 状态出现在玩家模型，但 MVP 推荐直接出局

如果 MVP 直接出局，`trapped` 暂时不应进入核心状态模型，否则测试和 UI 都会多处理一个未启用状态。

建议：

- MVP 删除 `trapped`。
- 后续“救援模式”再添加。

## 6. 范围压缩建议

建议把第一版收敛为：

### MVP 必须做

- 单机单服务房间。
- 2-4 个 Agent 对战。
- 格子移动。
- 水泡、爆炸、软墙、三类基础道具。
- 命中直接出局。
- 服务端权威 tick。
- 无 UI simulator。
- Web 前端观战。
- 本地策略调试面板。
- HTTP `/strategy` API。
- Agent 决策日志。

### MVP 暂缓

- OpenClaw 真接入。
- 人类手动玩家。
- 多在线房间。
- 踢泡、穿泡、遥控泡。
- 困住/救援模式。
- 完整复盘系统。
- 长期记忆。
- 强化学习。

OpenClaw 不应该被删除，只是不应成为核心引擎的前置依赖。

## 8. 第三方评审结论

从第三方视角看，这个方案有清晰的核心价值：玩家像教练一样通过对话塑造 Agent 战术，而不是自己操作角色。技术上也没有根本不可行点。

但它现在的最大问题是“边界不够窄”。如果按当前文档直接开工，很可能出现三个并行返工源：

- 游戏规则细节改动导致 Agent planner 重写。
- OpenClaw 接入提前进入导致调试链路复杂。
- 前端先做导致后续服务端协议反复调整。

推荐的修正路径：

1. 先锁 MVP：纯 Agent 对战、命中即死、无踢泡、单房间、Web 调试策略入口。
2. 先实现 `packages/core` 和 simulator。
3. 用模拟测试证明 Agent 不会频繁自杀。
4. 再接前端观战。
5. 最后把 OpenClaw 接到同一个策略 API。

这条路径能最大限度降低不确定性，同时保留原方案最重要的产品特色。

## 9. AgenTank 调研后的补充结论

调研 `https://agentank.ai/` 后，原评审的范围压缩建议仍然成立，但需要补一条更产品化的判断：第一版不能只证明规则引擎能跑，还要证明 Agent 迭代闭环成立。

AgenTank 的有效部分不是复古视觉本身，而是闭环完整：创建坦克、编辑逻辑、发布版本、真实对战、排行榜、历史对局、Agent Guide 和代码编辑器都围绕“持续改进同一辆坦克”组织。这个闭环给用户明确目标，也给 Agent 改进提供证据。

我们的方案应吸收三点：

- 第一版就要有策略版本、对局记录和轻量复盘，否则自然语言调教缺少反馈。
- 第一屏应是 Agent 工作台，而不是营销页或单纯游戏画面。
- 排行榜和历史记录即使是本地单服务版本，也应尽早出现。

但不建议复制 AgenTank 的 code-first 路线。我们的差异化仍应是 dialogue-first：用户通过自然语言改策略，界面突出危险图、计划路径、目标权重和决策解释。代码或策略 JSON 可以是高级模式，不应成为默认体验。

因此更新后的 MVP 应锁定为：单服务、2-4 个 Agent、无 UI simulator、Web 前端观战、站内策略面板、策略版本、对局记录、轻量复盘和简化排行榜。OpenClaw 真接入继续后置，复用同一个策略 API。
